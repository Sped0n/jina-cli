## Goal

Make the existing retry path more robust for transient Jina overloads such as HTTP 503, with a minimal change. Keep the current retry scope unchanged: retry Jina-hosted 5xx, 429, and transient network failures; do not broaden retries for non-Jina hosts.

## Recommended Approach

Tighten the retry-delay logic in `jina_cli/api.py` so the client more faithfully follows server retry guidance.

1. Update `_retry_wait(...)` to support both valid `Retry-After` formats:
   - delta-seconds (already supported today)
   - HTTP-date (currently ignored)
2. Keep the existing fallback behavior when `Retry-After` is missing or malformed:
   - use the current backoff schedule from `RETRY_BACKOFF`
   - cap all waits at `MAX_RETRY_WAIT`
3. Leave `_should_retry_status(...)` unchanged unless code review during implementation reveals a concrete bug.
4. Leave `jina_cli/utils.py` error messaging unchanged unless the implementation reveals that exhausted retries should surface a materially different exception shape.

## Files To Modify

1. `jina_cli/api.py`
   - update `_retry_wait(...)`
   - adjust any call sites only if needed to support parsing logic cleanly
2. `tests/test_local.py`
   - add unit coverage for `Retry-After` HTTP-date handling
   - keep existing retry-scope tests intact

## Implementation Notes

1. Prefer a minimal standard-library implementation for parsing HTTP-date values.
2. If the parsed HTTP-date is in the past, treat it as no extra delay beyond the normal backoff floor.
3. If parsing fails, silently fall back to the current backoff behavior.
4. Do not introduce a new retry framework, dependency, or cross-cutting abstraction for this change.

## Verification

1. Run focused unit tests for retry behavior:
   - `pytest tests/test_local.py -v`
2. Ensure these cases are covered:
   - `429` with numeric `Retry-After` still uses the larger of backoff and header value
   - `503` from a Jina host still retries
   - `503` from a non-Jina host still does not retry
   - HTTP-date `Retry-After` is honored when it points into the future
   - malformed HTTP-date falls back to normal backoff
   - very large retry delays remain capped at `MAX_RETRY_WAIT`
3. If there is an existing CLI or integration path that can be exercised cheaply, run the smallest relevant retry-related test subset after the unit tests.

## Expected Outcome

The CLI remains behaviorally the same in normal cases, but becomes more reliable when Jina responds with overload-style retry guidance, especially if the service returns `Retry-After` as an HTTP-date instead of a numeric delay.
